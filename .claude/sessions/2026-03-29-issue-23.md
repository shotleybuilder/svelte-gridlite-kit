### Key interfaces (defined in core)

```
QueryAdapter          — createLiveQuery(), execute(), executeCount(), destroy()
LiveQueryHandle       — subscribe(), refresh(), destroy()
SchemaProvider        — introspect() → ColumnMetadata[]
SuggestionProvider    — getDistinctValues(), getNumericRange()
```

## Decisions

- **Monorepo**: pnpm workspaces, single repo, shared test infra, atomic commits
- **Breaking changes OK**: single consumer (us), 0.x beta — no deprecation shims or backwards-compat props
- **Both adapters are separate packages**: PGLite (~3MB WASM) and TanStack DB are both heavy deps; core stays dependency-free
- **`db`/`table`/`source` props removed**: replaced by `adapter` prop, clean cut

## Phases

### Phase 1: Monorepo setup + adapter interfaces + PGLite adapter extraction
- Convert to pnpm workspaces monorepo (`packages/core`, `packages/pglite`)
- Define `QueryAdapter`, `LiveQueryHandle`, `SuggestionProvider`, `SchemaProvider` in core types
- Extract all PGLite-specific code into `packages/pglite`
- Remove `db`/`table`/`source` props from GridLite, replace with `adapter` prop
- Move `@electric-sql/pglite` from core dep to pglite adapter peer dep
- Tests: all existing tests pass under new structure, add adapter-level tests

### Phase 2: Decouple FilterBar/FilterGroup from PGLite
- Remove `db` prop from FilterBar and FilterGroup
- Replace direct `db.query()` calls with `adapter.getSuggestions()` / `adapter.getNumericRange()`
- Thread adapter through GridLite → FilterBar → FilterGroup
- Consolidate duplicate suggestion code between FilterBar and FilterGroup

### Phase 3: Decouple state persistence (migrations + views)
- Abstract `state/migrations.ts` and `state/views.ts`
- PGLite adapter bundles current behaviour
- TanStack DB adapter can use different persistence (or delegate to PGLite for storage)
- Decide: part of QueryAdapter or separate StateProvider interface?

### Phase 4: TanStack DB adapter
- `packages/tanstack-db` → `@shotleybuilder/gridlite-adapter-tanstack-db`
- Compile FilterNode[] → TanStack DB `where()` operators
- Compile SortConfig[] → `orderBy()`, GroupConfig[] → `groupBy()`
- Pagination, global search, JSONB → JS object access
- `@tanstack/db` as peer dependency
- Tests: equivalent IR inputs, same results as PGLite adapter

### Phase 5: Docs, skills, examples
- Update CLAUDE.md with adapter architecture
- Update skills files
- Add example route: `/examples/tanstack-db`
- Update README

## Notes
- `db.live.query()` is the hardest coupling — it's the reactive subscription backbone
- `query/builder.ts` is already pure (no db dependency) — stays in core, used by PGLite adapter
- FilterBar/FilterGroup have duplicate `db.query()` for suggestions — consolidate in Phase 2
- View persistence (Phase 3) flagged as optional/later in the issue
- Column-to-column comparison with INTERVAL offset (`valueColumn` + `intervalOffset`) has no TanStack DB equivalent — will need a custom comparator function in the TanStack DB adapter
- TanStack DB operators available: `eq`, `gt`, `gte`, `lt`, `lte`, `like`, `ilike`, `inArray`, `isNull`, `and`, `or`, `not`, `count`, `avg`, `sum`, `min`, `max`, `upper`, `lower`, `length`, `concat`, `coalesce`
- sertantai-legal is the only consumer — 3 pages use GridLite: browse, admin/lrt, admin/lat/queue (all on PGLite today)
